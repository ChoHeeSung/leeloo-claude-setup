# Model Selection Guideline (Skill Delegation Strategy)

Criteria for selecting a model when delegating skill-internal work to a sub-agent. `lk-commit` is the reference implementation (Task tool + `task_model`).

## Decision Criteria

| Type | Model | Criteria | Examples |
|------|-------|----------|----------|
| **Simple transform/formatting** | `haiku` | Deterministic input→output mapping, ≤1 reasoning step. Template fill-in, API result tabulation, YAML→SQL conversion | lk-commit (message generation), lk-its-ddl (DDL preview), lk-its-code (INSERT preview), lk-doc-parse, lk-n8n-node lookup, lk-bb-pr lookup |
| **Mid-complexity analysis** | `sonnet` | 2–3 reasoning steps, compare/summarize/decompose, mid-level domain knowledge | lk-code-review (solo), lk-todo (create decomposition), lk-doc-compare, lk-plan (Phases 3–4), lk-skill-create (Phase 3) |
| **Complex multi-step reasoning** | `opus` (keep main session) | Brainstorming, risk analysis, orchestration, vision integration, creative judgment | lk-plan (Phases 2 & 5), lk-plan-cross-review, lk-doc-pdf-extract, lk-commit main control |

## Delegation Pattern (Task tool)

When delegation is appropriate, write the corresponding step in SKILL.md as follows.

```markdown
### Step N. {task name} ({Haiku|Sonnet} Task)

The main session only composes the prompt and verifies the result. Actual work is delegated to a sub-agent.

**Agent tool call:**
- `subagent_type`: `task` (or `general-purpose`)
- `task_model`: `haiku` or `sonnet`
- `prompt`: self-contained prompt — specify context / input data / expected output format

**Result verification (main session):**
- [ ] Output format compliance
- [ ] No hallucination beyond the input data
- [ ] {Skill-specific check items}

**Fallback on quality miss:** regenerate directly in the main session (Opus).
```

## Cautions

- Do **not** delegate the main session's interactive AskUserQuestion, final confirmation, DB/file execution, or user-feedback incorporation (keep on Opus).
- Sub-agent prompts must be self-contained — do not assume the main session's chat history.
- Always specify a fallback to the main session on delegation failure or quality miss.

## When NOT to Delegate (gain < cost)

If any of the conditions below applies, the Sonnet delegation gain is smaller than the cost of context re-transmission and quality-loss risk, so **do not delegate**.

1. **Large context re-transmission with small per-call cost** — Have to send the full plan or a large doc to the sub-agent, but the generated output is short. Example: `lk-todo create` task decomposition — full plan re-transmission vs. 20–40 lines of JSON output.
2. **Within-session continuity with other Opus phases is critical** — Semantic consistency with content generated by Opus in adjacent phases is required. Example: `lk-plan` Phases 3–4 — Phase 2 (intent) and Phase 5 (writing) are Opus, so swapping the middle to Sonnet risks context discontinuity.
3. **Plan/design quality drives the project direction** — Quality regressions ripple to all downstream work. Keep on the main Opus session for consistency.

This still holds under a Max 5 plan environment. The Sonnet bucket is generous, but rework caused by quality loss can consume more of the Opus cap than the original delegation saved.
